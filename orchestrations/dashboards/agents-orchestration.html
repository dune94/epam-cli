LLM Mapping — Effort → Model Resolution
Each story in prd.json declares an effort field (low / medium / high). At runtime, claude.sh reads the story's effort and resolves the appropriate Claude model and --max-turns budget — ensuring cost efficiency without sacrificing quality for complex tasks.
Story Effort Claude Model Max Turns Typical Use Cases
low HAIKU 10 Docs, seed data, simple configs, unit test runner, CI tweaks
medium SONNET 30 Most feature stories — API endpoints, UI screens, reviews, migrations
high OPUS ∞ Complex architecture, cross-cutting concerns, skill assessments
Resolved inside claude.sh:   claude --model $(effortToModel "$effort") --max-turns $(effortToTurns "$effort") --output-format json --print  · Cost captured from JSON response → written to logs/phase-cost.jsonl
Execution Pipeline
🚀
run-agent-orchestration.sh
Master Orchestration Script — coordinates all steps below
Usagebash automation/scripts/run-agent-orchestration.sh --phase <PHASE> [--mode bash|hybrid]
Reads prd.json to discover stories for the given phase. Calls claude.sh per story (resolving effort → model → turns), manages git worktrees in HYBRID mode, and invokes check-phase-gate.sh at gates. All 8 steps below are orchestrated by this script.
BASH MODE HYBRID MODE
STEP 0.5
Pre-Phase Assessment
FULL TOOLS MAIN SONNET
Scriptclaude.sh --assessment pre --phase <PHASE>
Runs the agent-skill-assessment-agent before any stories execute. Inspects pending stories, detects skill gaps, and augments agent profiles in agents/profiles.json. Uses FULL TOOLS invocation (no --print) so the agent can read and write files directly. Runs on main branch.
STEP 1
Main Lane Setup
MAIN CLI
Per Storyclaude.sh --story <STORY_ID> --phase <PHASE>
Runs main-lane setup stories sequentially on the main branch — schema migrations (db-architect), seed data (mock-data-generator), and documentation (docs-engineer). Each invocation resolves the story's effort field to select the correct model and turn budget. Results are committed to the main branch before parallel work begins.
STEP 2
Create Worktrees
HYBRID ONLY
Commandgit worktree add wt-primary && git worktree add wt-independent
In HYBRID mode only — creates two isolated git worktrees from the current main branch HEAD. wt-primary hosts the PRIMARY lane; wt-independent hosts the INDEPENDENT lane. Both worktrees start with the latest schema and docs committed in Step 1. Skipped entirely in BASH mode.
STEP 3
Parallel Execution — Primary & Independent Lanes
PRIMARY INDEPENDENT CLI
Primaryclaude.sh --story <ID> --worktree wt-primary --phase <PHASE> & Independentclaude.sh --story <ID> --worktree wt-independent --phase <PHASE> &
Both lanes run simultaneously (HYBRID) or sequentially (BASH). Each story invocation is a separate claude.sh call that resolves the story's effort → model → turns before launching claude --print. In HYBRID mode, the coordination bus (agent-messages.jsonl) is active for cross-lane signalling.
PRIMARY Lane
git worktree: wt-primary
backend-engineer — API endpoints, business logic SONNET
pipeline-engineer — CI/CD, build automation SONNET
INDEPENDENT Lane
git worktree: wt-independent
frontend-engineer — Flutter UI screens SONNET
devops-engineer — Infra, Docker, env config SONNET
qa-engineer — Test specs, fixtures SONNET
STEP 4
Review & Merge Worktrees
MAIN CLI SONNET
Reviewclaude.sh --story <REVIEW_ID> --phase <PHASE> Mergegit merge wt-primary && git merge wt-independent
The review-agent runs on main, reads outputs from both worktrees, and validates code quality. After review, the orchestration script merges both worktrees back into main. Conflict resolution is handled by the review-agent if needed. Worktrees are removed after successful merge.
STEP 4.5
Unit Test Gate
GATE HAIKU MAIN
Scriptclaude.sh --story <UNIT_TEST_ID> --phase <PHASE>_unit Runnerdotnet test && flutter test
The unit-test-runner agent (HAIKU, low turns) executes all unit tests on the merged main branch. Both .NET backend tests and Flutter frontend tests must pass. This is a blocking gate — pipeline stops on failure.
✓ GO — all tests pass → continue to Step 5 ✗ NO-GO — test failure → pipeline halts, awaits fix
STEP 5
Phase Gate — Cost & Completion Check
GATE
Scriptcheck-phase-gate.sh --phase <PHASE>
Reads logs/phase-cost.jsonl and prd.json to verify two criteria: (1) all stories in the phase are marked completed: true, and (2) actual cost variance vs estimatedCost is within threshold. Decision written to logs/phase-gates.jsonl.
ok: variance < 30% warn: 30%–150% escalate: > 150%
✓ GO — all complete + cost ok → continue to Step 5.5 ✗ NO-GO — incomplete stories or cost escalation → halt
STEP 5.5
E2E Interstitial
GATE INDEPENDENT
Auto-detectif prd.json has phase <PHASE>_e2e → run-agent-orchestration.sh --phase <PHASE>_e2e
Automatically checks whether a corresponding E2E phase exists in prd.json (e.g. wearables_e2e after wearables). If found, the orchestration script recursively runs the E2E phase — using the qa-engineer to drive Playwright tests against the running stack. If no E2E phase exists, this step is silently skipped.
✓ E2E phase absent — step skipped, continue to Step 6 ✓ E2E phase present + all tests pass → continue ✗ E2E failures → pipeline halts
STEP 6
Post-Phase Assessment
FULL TOOLS MAIN SONNET
Scriptclaude.sh --assessment post --phase <PHASE>
Runs the agent-skill-assessment-agent post-phase with FULL TOOLS access. Analyses actual cost vs forecast from phase-cost.jsonl, identifies overruns, and corrects agentRole mismatches in pending stories within prd.json. Augments profiles for the next phase based on observed performance. Results visible in the PRD Viewer's Implementation Activity section.

Orchestration Modes
BASH Sequential Mode
All agents run sequentially in the main shell. No parallel worktrees — simpler, fully deterministic. Ideal for smaller phases or environments where git worktree isolation isn't needed.
  • Single shell process, single git branch
  • No worktree creation or teardown overhead
  • Coordination bus disabled (no inter-agent messages)
  • All agents run in main lane regardless of agentGroup
  • Cost captured per-story via --output-format json
HYBRID Parallel Mode
Primary and Independent agents run concurrently in isolated git worktrees, coordinated by the Main lane. Maximum throughput for multi-agent phases.
  • Two worktrees: wt-primary + wt-independent
  • Coordination bus active (agent-messages.jsonl)
  • Cross-worktree dependency checking via prd.json
  • Main lane gates progress between phases
  • Worktrees merged after phase gate passes
Key Features
🔄
Parallel Execution
Primary and Independent agents run simultaneously in isolated git worktrees. Hybrid mode maximises throughput; Bash mode provides deterministic sequential execution.
🎭
Role-Based Profiles
11 agent roles, each with a profile injected at invocation. Skills are tracked, augmented pre-phase, and assessed post-phase. Profiles stored in agents/profiles.json.
💰
Effort → LLM Mapping
Each story declares effort: low|medium|high. The pipeline maps this to model (Haiku / Sonnet / Opus) and max-turns (10 / 30 / unlimited) automatically via claude.sh.
🔗
Dependency Management
Cross-worktree dependency checking via prd.json ensures correct execution order. Stories declare dependencies and testFor links.
✅
Test Gate Pipeline
Step 4.5 runs unit tests before phase gate. Step 5.5 auto-detects <phase>_e2e phases and runs Playwright. Both gates block on failure.
📡
Coordination Bus
Hybrid mode only. Agents communicate via agent-messages.jsonl — approval messages, review feedback, alerts. Visible in the Agent Messages dashboard.
📊
Cost & Variance Tracking
Per-story cost captured via --output-format json. Phase gate checks variance against estimatedCost: ok <30% · warn 30–150% · escalate >150%.
🧠
Skill Assessment Loop
Pre-phase: detects skill gaps, augments profiles. Post-phase: analyses overruns, corrects agentRole mismatches in pending stories. Uses Full Tools invocation.
📋
Live Dashboards
Seven dashboards served at port 8090: Monitor · PRD Viewer · Agent Profiles · Orchestration · Test Plan · Agent Messages · Cost.
