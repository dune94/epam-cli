Estimation & CPA System

The EPAM CLI estimation system forecasts four metrics per story — AI execution minutes, cost (USD), tokens, and turns — before any agent is launched. Estimates flow through two sequential passes: a deterministic formula model followed by an LLM-based Contextual Purveyor Agent (CPA) review. The result is written to prd.json and used to gate phase approval and orchestration.

1. Two-Pass Pipeline

Running epam estimate (or epam phase run) executes both passes sequentially. The formula pass is deterministic and fast; the CPA pass is LLM-based and refines estimates with codebase context and risk reasoning.

1Formula Estimation

estimate-stories.sh

Reads prd.json story metadata and applies a complexity-weighted algebraic model. Outputs per-story predictions for minutes, tokens, turns, and cost using configurable constants calibrated from historical actuals.

  • Deterministic — no LLM calls
  • Runs in seconds for any number of stories
  • Uses per-tier human-to-AI ratios, cache models, pricing tables
  • Can be refined (--refine) from phase-cost.jsonl actuals

2CPA Pass

contextualize-stories.sh

For each story, retrieves relevant KB chunks via TF-IDF, then invokes the claude CLI with a rich context packet. The LLM reviews codebase signals, adjacent stories, and identified risks and returns a structured JSON adjustment.

  • LLM-based — uses claude --print (existing auth, no API key needed)
  • Blends CPA adjustment with formula estimate by confidence
  • Gates phase execution: PASS · REVIEW · BLOCK
  • Writes blended estimates to prd.json with --apply
epam estimate --phase finops --apply
# Runs: estimate-stories.sh → contextualize-stories.sh → writes prd.json

epam estimate --skip-cpa    # formula only, no LLM
epam estimate --strict      # halt on review gates (default: halt on block only)
epam estimate --refine      # calibrate constants from phase-cost.jsonl actuals

2. Pass 1 — Formula Estimation

Implemented in orchestrations/scripts/estimate-stories.sh. All math uses bc with 4-decimal precision.

2.1 Effort Tier Inference

Stories in prd.json carry a humanHours field (or estimatedHours as fallback) representing the developer's raw estimate in hours. This is mapped to one of three effort tiers which drive model selection and all subsequent constants:

TierhumanHoursDefault ModelHuman-to-AI Ratio
low≤ 2hclaude-haiku-*0.05
medium2h < h ≤ 6hclaude-sonnet-*0.08
high> 6hclaude-opus-*0.12

Note: If a story has an aiProvider or model field in prd.json, the estimator automatically resolves vendor-accurate pricing from src/billing/pricing.ts (the same registry used at runtime). The effort tier remains in use for human-to-AI ratios, turn rates, and token throughput — only the USD cost formula changes. Use --force-effort-tier to force the legacy Haiku/Sonnet/Opus proxy for all stories (useful for what-if comparisons).

2.2 Complexity Index C

The complexity index C is a dimensionless multiplier computed from story metadata. It amplifies or reduces the base human-to-AI conversion. C is bounded below by ~0.15 (health_check) and can exceed 1.5 for deeply interconnected high-complexity stories.

C = priority_weight × story_type_weight × dep_factor × skill_factor × loc_factor × file_factor × error_retry
FactorFormulaNotes
priority_weightlow/medium=1.0, high=1.1, critical=1.3Critical stories take longer due to correctness bar
story_type_weightimplementation=1.0, review=0.3, health_check=0.15Reviews and health checks are much cheaper
dep_factor1 + 0.05 × max(0, deps − 1)Each dependency beyond the first adds 5% overhead
skill_factor1 + 0.03 × max(0, skills − 2)Breadth beyond 2 skills adds cognitive load
loc_factor1 + 0.02 × max(0, totalLoc − 200) / 100Larger files = more context to load and reason about
file_factor1 + 0.05 × max(0, files − 2)Multi-file coordination overhead
error_retryProduct of applicable risk multipliersHTTP/MCP/SSE: ×1.20; agent-orchestration: ×1.25; >3 files: ×1.15

2.3 Turn & Minute Computation

Turn computation is split into two stages to cleanly separate work complexity from execution speed. This allows the same model to be used regardless of whether agents run autonomously (fast) or interactively (slow).

1
Conceptual complexity-minutes — how much work the story represents, independent of speed:
complexity_minutes = humanHours × H2A_ratio × 60 × C
2
Turn count — derived using interactive (human-step) rates and an autonomous multiplier:
turns = ceil(complexity_minutes / interactive_mpt × AUTONOMOUS_TURN_MULTIPLIER)
Autonomous agents iterate more aggressively; multiplier = 1.5 in autonomous mode.
3
Wall-clock AI minutes — actual elapsed time from turns × execution mode rate:
ai_minutes = max(AI_MINUTES_FLOOR, turns × execution_mpt)
Floor = 0.50 min — irreducible overhead for CLI startup and context loading.
Modemin/turn (low)min/turn (med)min/turn (high)
autonomous (default)0.080.100.15
interactive0.501.001.50

Set EXECUTION_MODE=interactive env var to switch modes.

2.4 Token Estimation

Tokens are estimated from AI minutes and a tokens-per-minute rate, adjusted for file count and context accumulation (tokens grow super-linearly with turns due to growing conversation history).

tokens = ai_minutes × tokens_per_min[effort] × file_token_factor × ctx_growth_factor file_token_factor = 1 + 0.1 × max(0, file_count − 1) ctx_growth_factor = 1 + 0.12 × ln(turns) [capped at 1 for single-turn stories] # Context growth examples: 1 turn=1.00x, 10 turns=1.28x, 30 turns=1.41x, 60 turns=1.49x
Effort TierTokens/minInput ratioOutput ratio
low (Haiku)8,00090% (health_check), 75% (impl)10% / 25%
medium (Sonnet)12,00092% (review), 75% (impl)8% / 25%
high (Opus)18,00075% (impl)25%

Input/output ratio varies by storyType: implementation generates more output (code); reviews are read-heavy.

2.5 Cache Model

The Claude API caches prompt tokens automatically. Stories benefit progressively more from the cache as they advance through a phase (later stories warm-hit the shared system prompt and project context from earlier stories' runs).

Position in PhaseCache Hit RatioNotes
1st story0.30Cold-start baseline (CLI auto-caches ~30K tokens of project context)
2nd story0.50Prior story's context now hot
3rd story0.60Cumulative warmup
4th+ stories0.70Steady-state cache saturation
+Same agent role+0.10Consecutive stories using the same agentRole share more context
+>50% file overlap+0.15Stories touching the same files benefit from cached file reads
+Some file overlap+0.08Partial file overlap

Cache ratio is capped at 0.85. The first story in every phase pays a one-time CACHE_PRIMING_COST of $0.06 for system prompt + project context cache creation.

2.6 Cost Formula

input_tokens = tokens × input_ratio output_tokens = tokens × output_ratio cache_tokens = input_tokens × cache_ratio uncached_in = input_tokens − cache_tokens cost = (uncached_in / 1M) × price_input + (cache_tokens / 1M) × price_cached + (output_tokens / 1M) × price_output + (cache_priming_cost if position == 1)

Pricing is resolved per story using the following priority chain:

PriorityConditionPrice source
1Story has .model set in prd.jsonsrc/billing/pricing.ts — exact model rate (e.g. gpt-4.1 → $2.00/$8.00 per M)
2Story has .aiProvider set (no model)Provider default model → pricing.ts lookup (e.g. qwen → qwen/qwen3-235b-a22b)
3Neither set, or model not in registryEffort-tier proxy below (Haiku / Sonnet / Opus rates)

Subscription-billed providers (copilot, cursor) are emitted as $0.00 (subscription) — no per-token cost is fabricated. Pass --force-effort-tier to bypass the vendor lookup and use the effort-tier proxy for all stories.

2.7 Uncertainty Ranges

Each estimate includes a ±range reflecting modelling uncertainty. The range widens for more complex stories and more turns:

range_factor = 0.15 + 0.03 × ln(C × turns + 1) [capped at 0.50] minutes_low = ai_minutes × (1 − range_factor) minutes_high = ai_minutes × (1 + range_factor × 1.5) cost_low = cost × (1 − range_factor) cost_high = cost × (1 + range_factor × 1.5)

2.8 Bayesian Calibration (--refine)

When --refine is passed and phase-cost.jsonl contains completed story records, the model updates its constants from observed actuals. This narrows forecast uncertainty over time.

Three constants are calibrated per effort tier:

  1. HUMAN_TO_AI_RATIO — sum(elapsed_minutes) / sum(humanHours × 60)
  2. TOKENS_PER_MIN — sum(tokens_in + tokens_out) / sum(elapsed_minutes)
  3. MIN_PER_TURN — sum(elapsed_minutes) / sum(turns), clamped [0.02, 2.0]

Posteriors are stored in orchestrations/logs/calibration.json grouped by effort:storyType category (e.g. low:health_check, medium:implementation). Each category tracks: n, mean_minutes, mean_cost, mean_tokens, mean_turns, variance, and updatedAt.

Convergence: The model reaches ±10% precision on typical categories after approximately 8–12 completed stories in that category. Running --refine after every phase is strongly recommended.

3. Pass 2 — Contextual Purveyor Agent

Implemented in orchestrations/scripts/contextualize-stories.sh and orchestrations/scripts/lib/cpa-inference.js. The CPA is the AI-augmented layer: it reviews each story's formula estimate in full context and either confirms it, adjusts it, or flags it for human review.

3.1 TF-IDF Knowledge Base

Before invoking the LLM, the CPA retrieves the most relevant KB chunks for each story. The knowledge base indexes all .md and .txt files in orchestrations/agents/ (KB.md, AGENTS.md) plus extra docs: INSTRUCTIONS.md, estimation.md, README.md.

Retrieval is performed by tfidf.js:

node tfidf.js \
  --kb-dir orchestrations/agents \
  --query  "TypeScript OAuth device flow JWT" \
  --top    5 \
  --extra-docs orchestrations/estimation.md,orchestrations/INSTRUCTIONS.md

The query is built from the story's title + description + required skills. Top-5 chunks are included in the LLM prompt along with their relevance scores.

3.2 CPA Inference (cpa-inference.js)

cpa-inference.js receives a JSON payload on stdin and calls the claude CLI with --print --dangerously-skip-permissions. It uses the existing Claude Code authentication — no separate API key is required.

Input payload structure:

{
  "story":           { id, title, description, priority, storyType, effort, ... },
  "kbChunks":        [{ source, score, chunk }],       // TF-IDF results
  "codebaseSignals": { totalLoc, fileCount, importCount, filesExist, fileSnippets },
  "formulaEstimate": { aiMinutes, cost, tokens, turns },
  "adjacentStories": [{ id, title, effort, status }],  // other stories in same phase
  "systemPrompt":    "string"                          // from orchestrations/prompts/cpa-system.md
}

Output (returned as JSON):

{
  "confidence":           0.82,        // 0.0–1.0 — how certain CPA is of its review
  "complexityAdjustment": 1.15,        // multiplier on formula estimate (clamped ±30%)
  "adjustedEstimate": { aiMinutes, cost, tokens, turns },
  "riskFlags":        ["MCP integration untested", "3rd-party OAuth dependency"],
  "missingKbCoverage":["SSE streaming", "token refresh flow"],
  "citedSources":     ["KB.md#oauth-flow"],
  "reasoning":        "Story involves non-trivial OAuth... adjusted +15% for...",
  "_metrics":         { latencyMs, tokensIn, tokensOut, tokenEfficiency }
}

3.3 Confidence Blending

The CPA review is blended with the formula estimate based on the returned confidence value. Higher confidence = more CPA weight:

Confidence RangeBlending Rule
≥ 0.75 (BLEND_HIGH)Use CPA estimate fully
0.50 – 0.74Linear interpolation: blend = (confidence − 0.50) / (0.75 − 0.50)
0.35 – 0.49Keep formula + 20% uncertainty markup; gate=REVIEW
< 0.35 (GATE_BLOCK)Gate=BLOCK — CPA cannot make a reliable assessment
blended = formula × (1 − blend_weight) + cpa × blend_weight

3.4 Gate Logic

After blending, each story is assigned a gate outcome. The phase cannot proceed past a BLOCK gate without manual override; REVIEW gates can continue by default but halt with --strict.

✓ PASS — No issues found, estimates accepted
⚠ REVIEW — Risk flags or low confidence; manual check recommended
✗ BLOCK — CPA cannot proceed; story must be revised
ConditionGate
confidence ≥ 0.50 and riskFlags.length ≤ 3PASS
confidence 0.35–0.49, OR riskFlags.length > 3REVIEW
confidence < 0.35BLOCK

Override BLOCK gates: SKIP_CPA=1 epam orchestrate --phase finops

Halt on REVIEW gates: epam estimate --strict or STRICT_CPA=1

3.5 Guardrails

To prevent runaway LLM adjustments, all CPA outputs are clamped before application:

  • complexityAdjustment clamped to [0.70, 1.40] — CPA can reduce by at most 30%, increase by at most 40%
  • adjustedEstimate fields clamped independently: each field's ratio vs. formula clamped to [0.70, 1.40]
  • turns — absolute floor of 3 turns (CPA cannot produce fewer than 3 turns)
  • confidence clamped to [0.0, 1.0]

If the claude CLI fails or returns unparseable output, the CPA falls back to the formula estimate unchanged (_inferenceSkipped: true in the log) and continues. No story is ever left without an estimate.

5. Provider Layer

The orchestration system supports multiple AI providers per phase or per story. Four providers — copilot, openai, qwen, and cursor — run via the epam run --json interface. Existing providers (claude, codex, codemie-claude) retain their direct CLI invocation paths.

5.1 Provider Routing

run-agent-orchestration.sh selects the entry-point script using the EPAM_ORCHESTRATION_PROVIDER environment variable. Each script is a thin wrapper that sets EPAM_CLI and execs the master claude.sh agent loop — no logic is duplicated.

EPAM_ORCHESTRATION_PROVIDEREntry scriptInvocation method
(default)claude.shclaude CLI direct
codemie-claudecodemie-claude.shcodemie-claude CLI direct
copilotcopilot.sh → claude.shepam run --provider copilot --json
openaiopenai.sh → claude.shepam run --provider openai --json
qwenqwen.sh → claude.shepam run --provider qwen --json
cursorcursor.sh → claude.shepam run --provider cursor --json

Inside claude.sh, per-story routing is handled by provider_to_cli() which reads the story's aiProvider field and returns the correct binary. The EPAM_CLI variable (default: epam) is the binary used for all epam-run providers and can be overridden for testing.

provider_to_cli("copilot") → $EPAM_CLI (epam run --provider copilot ...) provider_to_cli("openai") → $EPAM_CLI (epam run --provider openai ...) provider_to_cli("qwen") → $EPAM_CLI (epam run --provider qwen ...) provider_to_cli("cursor") → $EPAM_CLI (epam run --provider cursor ...) provider_to_cli("codex") → codex provider_to_cli("*") → $CLAUDE_CMD (claude)

5.2 Per-Story aiProvider & model Fields

Each story in prd.json can carry an aiProvider field and an optional model field. These drive both execution routing and cost estimation:

FieldEffect on executionEffect on cost estimate
aiProviderRoutes story to the matching CLI via provider_to_cli()Triggers provider-default model lookup in pricing.ts
modelPassed as --model to the CLI invocation; overrides effort-tier defaultDirect lookup in pricing.ts — most precise rate

For epam-run providers, resolve_model_from_story() reads the story's .model field directly from prd.json and overrides the effort-tier default before invocation. This means a story with "aiProvider": "openai", "model": "gpt-4.1-mini" will both run on gpt-4.1-mini and be costed at gpt-4.1-mini rates ($0.40/$1.60 per M).

5.3 Cost Normalization

epam run --json outputs camelCase fields. claude.sh's normalize_provider_json() maps these to the snake_case schema expected by append_cost_record and phase-cost.jsonl:

epam run --json output: Normalized (orchestration schema): ───────────────────────────────────────────────────────── cost_usd → total_cost_usd usage.inputTokens → usage.input_tokens usage.outputTokens → usage.output_tokens result → result (preserved)

This normalization ensures all providers feed the same cost tracking pipeline regardless of their output format. The raw file is preserved alongside the normalized file for debugging.

5.4 Story Selection Rules

Story selection differs between execution and estimation modes:

ModeFilter appliedRationale
Execution
run-agent-orchestration.sh		 completed == false Only unfinished stories are run. The completed boolean is the sole gate — there is no status string field check. Stories already marked completed: true are silently skipped.
Estimate
estimate-stories.sh		 No status filter All stories in scope are estimated regardless of completion. This is required so --refine can calibrate constants from completed story actuals, and so forecast totals include future stories.
Dry-run
--dry-run flag		 completed == false (same as execution) Story selection and plan display run fully; execution is skipped after plan output.

Infra test gate: For any phase other than infra_test, all SP-T0x stories in the infra_test phase must be completed: true before execution proceeds. This gate runs before story selection and cannot be bypassed by --dry-run.

5.5 Zero-Token Testing

The EPAM_CLI variable is the injection point for testing epam-run provider flows without spending tokens. Set it to the mock script to intercept all epam-run invocations: 

EPAM_CLI=orchestrations/scripts/test/mock-epam-run.sh \
  bash orchestrations/scripts/test/test-epam-providers.sh
# → 19/19 tests pass; zero tokens consumed

The mock returns a canned epam run --json response and logs --provider / --model flags to $MOCK_LOG for assertion. Tests cover: provider routing, normalize transform, cost extraction, and model resolution.

6. Integration with CLI Commands

CommandWhat it runsKey flags
epam estimate estimate-stories.sh → contextualize-stories.sh --phase, --apply, --refine, --skip-cpa, --strict, --json, --reconcile, --force-effort-tier
epam phase run estimate → CPA → orchestrate (full pipeline) --phase, --require-approval, --skip-cpa, --strict-cpa, --dry-run
epam phase approve Records human approval for a phase snapshot --phase
epam orchestrate run-agent-orchestration.sh (skips estimate) --phase, --mode bash|hybrid, --skip-cpa, --strict-cpa, --provider copilot|openai|qwen|cursor

Typical workflow:

# 1. Get formula estimates (dry run — see numbers, no writes)
epam estimate --phase finops

# 2. Run full estimate + CPA, write results to prd.json
epam estimate --phase finops --apply

# 3. What-if comparison: force effort-tier pricing regardless of aiProvider/model
epam estimate --phase finops --force-effort-tier

# 4. Approve and run the phase
epam phase approve --phase finops
epam phase run --phase finops --require-approval

# 5. Run via a specific provider (copilot/openai/qwen/cursor)
EPAM_ORCHESTRATION_PROVIDER=openai epam orchestrate --phase finops

# 6. After phase completes, refine constants for next run
epam estimate --refine    # reads actuals from phase-cost.jsonl
