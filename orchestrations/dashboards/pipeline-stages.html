Overview

This document is the authoritative reference for the EPAM CLI multi-agent orchestration pipeline — the automated system that takes a PRD (product requirements document) from raw stories through specification, cost estimation, parallel implementation, quality assurance, post-execution assessment, and documentation generation. The pipeline is driven by run-agent-orchestration.sh and run-doc-orchestration.sh, coordinating 36+ specialized AI agents across 6 sequential stages, each with distinct tooling, gating policies, and parallelization strategies. Agents are the focal point: every stage section below details which agents participate, what ReAct loop or inference mode they operate in, which built-in tools (ReadFile, WriteFile, Bash, Search, ListFiles, FetchUrl) and external tooling (tsc, vitest, Knip, jscpd, Semgrep, ast-grep, Madge, TypeDoc, Vale, Mermaid.js, git-cliff, Markprompt) they leverage, and what data files they read from and write to.

The pipeline supports two orchestration modes — Bash (default, script-driven) and Hybrid (Bash + MCP message bus for inter-agent coordination) — and executes implementation stories across three parallel lanes (Main, Primary, Independent) using isolated git worktrees. Quality is enforced through 15+ gate checkpoints: a CPA estimation gate, code review escalation gates, three cascading QA gate phases (each running two agents in parallel), a unit test hard gate, a final phase gate with cost variance thresholds, and documentation review gates with a 3-cycle feedback loop. Cost tracking spans the entire pipeline via phase-cost.jsonl actuals, Langfuse LLM tracing, and CPA confidence-weighted estimate blending. All agents emit structured events to agent-activity.jsonl for unified observability. The Complete Agent Registry at the bottom of this page provides a single cross-reference of every agent, its source definition, stage participation, model assignment, tools, and outputs.

Specification
SEQUENTIAL REACT LOOP GATE
▼
Flow
  • Entry: spec-mode-runner.js (Step 0)
  • Pre-Spec Grooming: grooming-coordinator routes stories to readiness-checker (INVEST/DoR) and dedup-detector (Simili-Bot)
  • Stories with readinessScore < 60 flagged as not-ready-for-spec (skip unless FORCE_SPEC=1)
  • Snapshots current prd.json as baseline
  • Coordinator reads all phase stories and decides which spec personas to launch
  • OpenSpec agent elaborates acceptance criteria, proposes story splits, adds technical notes
  • Speckit agent reviews OpenSpec output, adds testability and security criteria
  • Edits applied back to prd.json (acceptance criteria, splits, notes)
  • Skip grooming: SKIP_GROOMING=1  |  Skip spec: EPAM_SPEC_MODE=0
ReAct Loops
  • Grooming coordinator: Single-turn inference — routes stories to grooming agents, aggregates results
  • Readiness-checker: Single-turn inference — INVEST scoring + DoR checklist per story
  • Dedup-detector: Single-turn inference — pairwise semantic similarity + Simili-Bot CLI
  • Spec coordinator: Single-turn inference — reads stories, outputs agent assignment JSON
  • OpenSpec: Multi-turn ReAct — reads each story, reasons about gaps, outputs elaborated criteria
  • Speckit: Multi-turn ReAct — reads OpenSpec output, reasons about testability/security, outputs complement criteria
  • Each agent invoked via claude.sh with full tool access (ReadFile, WriteFile, Bash, Search)
Parallelization
  • None — strictly sequential
  • Coordinator → OpenSpec → Speckit (each reads prior output)
  • Each agent must complete before the next starts to avoid conflicting PRD edits
Gates
  • No hard gate — spec is optional (skip with env var)
  • Implicit validation: coordinator may decide "none" if stories are already well-specified
  • Outputs archived to orchestrations/logs/spec-runs/<run>/ for audit
Skills Injection
  • Agent profiles loaded from profiles.json
  • Coordinator prompt includes full story list + agent roster
  • OpenSpec/Speckit prompts include their role definition + target story context
  • KB.md injected for domain context
Cost Tracking
  • Each agent invocation captures tokens + cost from claude.sh JSON response
  • Written to phase-cost.jsonl with story_id: spec-pass
  • Spec pass cost visible in phase-cost-monitor dashboard
CPA
  • Not applied at this stage
  • CPA runs after spec, using the elaborated stories as input

Estimation
SEQUENTIAL REACT LOOP CPA GATE
▼
Flow
  • Entry: estimate-stories.sh (30 KB) + contextualize-stories.sh (32 KB)
  • Phase A: Formula-based estimation — complexity-weighted model computes tokens, cost, hours per story
  • Phase B: CPA (Contextual Purveyor Agent) pre-pass — TF-IDF KB retrieval + LLM-based blended estimation
  • Phase C: Pre-phase skill assessment — team-lead detects skill gaps in agent profiles
  • Phase D: Hybrid coordination (hybrid mode only) — identifies cross-lane dependencies
  • Skip CPA: SKIP_CPA=1  |  Strict CPA: STRICT_CPA=1
ReAct Loops
  • Formula estimation: No ReAct — pure shell computation (effort → tokens/cost/hours)
  • CPA agent: Multi-turn ReAct — reads KB.md via TF-IDF, reasons about story context, outputs blended estimates
  • Skill assessment: Single-turn inference — reads profiles + story requirements, outputs augmentation recommendations
  • Hybrid coordinator: Single-turn inference — reads story graph, outputs dependency + plan recommendations
Parallelization
  • None — strictly sequential pipeline
  • Formula → CPA → Skill Assessment → Hybrid Coord
  • Each step depends on the prior step's output
Gates
  • CPA 3-tier gate:
  • PASS (exit 0) — all stories cleared, proceed
  • REVIEW (exit 2) — elevated risk, auto-approved with warning
  • BLOCK (exit 3) — cannot proceed, requires manual resolution
  • Infra gate: Blocks non-infra phases if infra_test stories incomplete
  • STRICT_CPA=1: Halts on REVIEW gates (default: continue)
Skills Injection
  • Step 0.5: Team-lead reads prd.json story requiredSkills
  • Compares against agent profile text in profiles.json
  • Augments profiles with missing skills (append-only)
  • Audit trail: profiles-audit.jsonl
  • Backup: profiles.json.original (created on first run)
Cost Tracking
  • Formula estimates: Written to prd.json per story (estimatedCost, estimatedTokens, estimatedHours, estimatedTurns, estimatedAiMinutes)
  • CPA blending: Confidence-weighted interpolation:
    • ≥ 0.75 confidence: Trust CPA 100%
    • 0.50–0.75: Interpolate formula + CPA
    • < 0.50: Keep formula + 20% uncertainty markup
  • Historical calibration: CPA reads phase-cost.jsonl actuals
CPA (Contextual Purveyor Agent)
  • Core function: KB-grounded estimate refinement
  • TF-IDF sources: KB.md, INSTRUCTIONS.md, estimation.md, README.md
  • Reads: Formula estimates + historical actuals
  • Outputs: cpa-review.jsonl (gate decisions), reconciliation reports
  • Blending formula: confidence-weighted interpolation between formula and CPA estimates

Execution / Implementation
PARALLEL REACT LOOP BASH HYBRID
▼
Flow
  • Entry: run-agent-orchestration.sh (67 KB)
  • Step 1: Main-lane stories (sequential) — agentGroup=main
  • Step 2: Create worktrees for primary + independent lanes
  • Step 3a/3b: Primary + Independent lanes execute in parallel (background PIDs)
  • Step 3.1: Worktree health check + auto-commit
  • Step 3.2: Merge worktrees back to main (--no-ff, conflict detection, tsc check)
  • Step 3.5: Post-parallel skill assessment (cost variance, role recommendations)
  • Step 3.6: Team-lead code review (max 3 iterations per story, escalation on failure)
ReAct Loops
  • Story agents: Full ReAct loop per story via claude.sh
  • Effort-based turn budgets: LOW 10 MED 30 HIGH ∞
  • Each agent has ReadFile, WriteFile, Bash, Search, ListFiles tools
  • Code review cycle: Up to 3 ReAct iterations per story (review → feedback → fix → re-review)
  • Skill assessment: Single-turn inference (no ReAct loop)
Parallelization
  • 3-lane architecture:
  • MAIN Sequential — runs first, completes before worktrees
  • PRIMARY Parallel — background PID, isolated worktree branch wt-primary
  • INDEPENDENT Parallel — background PID, isolated worktree branch wt-independent
  • Primary + Independent run concurrently via wait $PID
  • Within each lane, stories execute sequentially
Gates
  • Worktree health check: Validates clean state before merge (exit 0/1/2)
  • Merge gate: Conflict detection, tsc --noEmit post-merge (exit 0/1/2/3)
  • Code review gate: Team-lead reviews each story — escalation (exit 2) blocks pipeline
  • Auto-commit: AUTO_COMMIT=true stages src/lib/packages and commits
Skills Injection
  • Agent profiles from profiles.json injected as system prompt per story
  • Story-specific technicalNotes appended to prompt context
  • KB.md injected for shared domain knowledge
  • Post-parallel assessment (Step 3.5): Detects skill mismatches and recommends role changes for future stories
Cost Tracking
  • Per-story: claude.sh captures JSON response → tokens_in, tokens_out, cost_usd, elapsed_minutes
  • Atomic append: flock-protected writes to phase-cost.jsonl
  • Variance analysis: Step 3.5 compares actual vs forecast per story
  • Lane tagging: Each record includes lane (main/primary/independent)
  • Langfuse: TracedProvider captures detailed LLM traces (tokens, latency, tool calls)
CPA
  • Not active during execution
  • CPA estimates from Stage 2 used as forecast_hours baseline for variance analysis
  • Actuals feed back into CPA calibration for future phases

Quality Assurance
PARALLEL REACT LOOP 3 CASCADE GATES
▼
Flow
  • 3 cascading gate phases — each must pass before the next runs
  • Phase A: SAST + Spec Compliance (parallel)
  • Phase B: Code Review + Mutation Testing (parallel, if A passed)
  • Phase C: Fuzz Testing + Performance (parallel, if A+B passed)
  • Step 4.5: Unit test gate (tsc --noEmit + vitest run) — non-negotiable
  • Skip all QA: SKIP_TESTING_GATES=true
ReAct Loops
  • All 6 QA agents: Multi-turn ReAct with tool access
  • Each agent reads source code, analyzes patterns, outputs structured JSON verdict
  • sast-sentinel: Parses tsc output + scans security patterns
  • spec-validator: Maps AC → implementation, classifies met/partial/unmet
  • review-ranger: Deep diff analysis (complexity, duplication, API drift)
  • mutant-hunter: Proposes mutations, classifies killed/survived/no-coverage
  • fuzz-weaver: Derives input domains from types, proposes fast-check cases
  • perf-sentinel: Algorithmic complexity, memory, async pattern analysis
Parallelization
  • Intra-phase: 2 agents run in parallel within each gate phase
  • Inter-phase: Phases cascade sequentially (A → B → C)
  • Phase A: sast-sentinel ‖ spec-validator
  • Phase B: review-ranger ‖ mutant-hunter
  • Phase C: fuzz-weaver ‖ perf-sentinel
  • Both agents in each phase must complete; pipeline waits for both
Gates
  • Phase A gate: Both verdicts must be != "fail" to proceed to B
  • Phase B gate: Both verdicts must be != "fail" to proceed to C
  • Phase C gate: Both verdicts checked; failures logged
  • Verdict tiers: PASS WARN FAIL
  • Unit test gate (4.5): tsc --noEmit + vitest run — hard block, non-negotiable
  • Thresholds: spec-validator ≥80% met, mutant-hunter ≥70% score, fuzz 0 vulns, perf 0 blockers
Skills Injection
  • Each QA agent has a specialized system prompt from profiles.json
  • Prompts include: focus areas, output JSON schema, verdict criteria, threshold definitions
  • Additional QA agents: hygiene-sentinel (Knip), design-sentinel (jscpd/Semgrep), pattern-sentinel (ast-grep/Madge)
Cost Tracking
  • Each QA agent invocation logged to phase-cost.jsonl
  • Duration tracked per agent: duration_ms in testing-gates.jsonl
  • QA cost visible as separate line items in phase-cost-monitor dashboard
  • Langfuse traces capture per-agent token usage and latency
CPA
  • Not active during QA stage
  • QA gate costs feed back into CPA calibration for future estimation passes
