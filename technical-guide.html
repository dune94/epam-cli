EPAM CLI Technical Guide

Comprehensive reference for the EPAM CLI runtime, agent architecture, orchestration scripts, dashboards, and operational tooling.

1. System Overview

Layer Responsibilities Key Entry Points
CLI Runtime Command routing, auth, chat/run commands src/index.ts, src/cli/index.ts
Agent Core Conversation loop, tools, memory injection, failover src/agent/AgentRunner.ts, src/providers/ProviderChain.ts
Context & Storage Session JSONL, Redis export/import, memory loader src/context/*.ts, src/memory/
Orchestration PRD-driven phases, multi-agent runs, behavioral contracts, brownfield ingestion, webhook triggers orchestrations/scripts/run-agent-orchestration.sh
Dashboards Eleventy builds, BrowserSync, provider/model filters, deployment orchestrations/dashboards/*

2. CLI Architecture

2.1 Entry & Commands

src/index.ts bootstraps Commander, injects chat when users only pass global flags, and parses asynchronously with helpful error output. src/cli/index.ts registers every command module so interactive and scripted flows reuse the same plumbing (auth, config, provider selection).

2.2 Config & Auth

  • ConfigResolver merges flags, EPAM_* env vars, project (.epam/settings.json), and global config (~/.epam/config.json), recording the model selection source for audit trails.
  • AuthManager runs device/browser login flows, persists secrets in the OS keychain, and backs epam provider * commands.
  • v1 Auth bridge model (DEC-005): User-managed API keys stored via KeychainKeyStore. Manual entry via epam provider login <provider> or EPAM_API_KEY_* env vars. Browser PKCE for Codemie. No auto-provisioned brokered keys in v1 — deferred to v2+. See .epam/decisions.jsonl DEC-005.

2.3 Session Lifecycle

  • SessionStore writes turns to .epam/sessions/<ulid>.jsonl for resume/fork/import flows.
  • RedisSessionStore (via EPAM_REDIS_URL) powers /share and /handoff by storing serialized bundles with a 7-day TTL.

2.4 REPL UX

Repl coordinates provider failover, slash commands, streaming output, and history. RawInputBox draws the prompt, supports bracketed paste, and keeps cursor math accurate for multi-line input.

2.5 Agent Execution

  1. chat builds the system prompt (constraints + consultation context).
  2. AgentRunner maintains conversation state and token ceilings.
  3. MEMORY.md injection (EPAM-039): MemoryLoader reads MEMORY.md from the project root, resolves linked memory files via MemoryImportResolver, and injects a summarized block into the system prompt on the first run() call. Memory reloads on /compact. Lazy injection avoids async I/O in the synchronous constructor.
  4. Tool calls go through ToolRunner (Read/Write/List/Search/Fetch/Bash).
  5. RalphWiggumLoop retries fragile bash steps automatically.
  6. AuditorRunner feeds lint/test findings back into the chat.

2.6 Provider Chain

ProviderChain spins up slots for Anthropic, OpenAI, Gemini, Copilot, Codex, Cursor, Qwen, Codemie, and Proxy providers. It tracks slot health, classifies errors (retry vs. failover), and emits events consumed by the REPL. Backups follow the policy documented in plans/orchestration-failover-plan.md so CLI and orchestration share the same strategy.

2.7 Batch Commands

epam run executes a single prompt non-interactively but still uses the agent pipeline. estimate, orchestrate, phase, report, etc. wrap orchestration scripts to keep tooling consistent.

3. Orchestration Architecture

3.1 Story Data

  • orchestrations/prd.json — backlog, phases, quotas.
  • orchestrations/agents/profiles.json — persona prompts.
  • AGENTS.md, KB.md — learned behaviors & knowledge base.

3.2 Script Flow

  1. Bootstrap paths, pick provider wrapper, ensure logs.
  2. Start Eleventy watcher (unless EPAM_DASH_AUTO_SERVE=0).
  3. Run specification pre-pass (spec-mode-runner.js) — snapshot prd.json, assign OpenSpec/Speckit per story, apply acceptance criteria edits or splits, and log deltas to logs/spec-phase.jsonl (disable with EPAM_SPEC_MODE=0).
  4. Resolve phase + orchestration mode via phasesConfig.
  5. Run CPA + execution-plan jq pipelines.
  6. Forecast token ceilings, precompute provider/model backups.
  7. Create git worktrees per lane and seed manifests.
  8. Launch primary/independent agents via provider wrapper scripts.
  9. Update dashboards + logs, trigger provider-cutover.sh as needed.
  10. Run reviews, tear down worktrees, stop watcher.

3.3 Utilities

Wrapper scripts (claude.sh, copilot.sh, openai.sh, etc.) normalize provider CLIs. Supporting helpers include contextualize-stories.sh, estimate-stories.sh, team-lead-review.sh, update-monitor.sh, and provider-cutover.sh.

3.6 Behavioral Contracts (GAP-P8 / GAP-P10)

  • Static constitution (GAP-P8): AGENT_CONSTITUTION in claude.sh — four non-negotiable rules (filesystem boundary, AC verification, protected paths, credential safety) injected into every agent invocation.
  • Dynamic augmentation (GAP-P10): resolve_dynamic_constitution() reads .epam/constitution-rules.json at invocation time and appends rules whose match.skills or match.agentRole overlap the story's metadata. Rules reset per story. Absent file = identical to GAP-P8 behavior.

3.7 Brownfield Context Ingestion (GAP-P9)

  • Activated by brownfield.repoRoot in the PRD. lib/brownfield-context.js runs at CPA time, chunking the target repo via git ls-files and scoring with TF-IDF.
  • Stage 2 reads .epam/brownfield/jira.json and confluence.md from the target repo. When JIRA_URL/JIRA_EMAIL/JIRA_TOKEN are set, fetches live Jira issues; falls back to stubs silently.

3.8 External Event Triggers (GAP-P2)

  • control-plane.js exposes POST /webhook/jira (when JIRA_WEBHOOK_SECRET is set). Payloads are HMAC-verified, adapted by lib/jira-adapter.js, and queued by lib/webhook-queue.js with a 45-second debounce window. Urgent-labelled events bypass the window.
  • jira-writeback.sh posts milestone updates (spec, CPA, story-complete, review-done) back to Jira via lib/jira-client.js. No-ops when JIRA_URL is unset.

3.9 LLM Topology Routing (GAP-P11)

  • Before execution, run-agent-orchestration.sh calls lib/topology-router.js with story metadata. The router makes a single Haiku tool-call returning {topology, reason, source} via Claude's structured output (select_topology tool schema).
  • single — lone story or tight coupling: collapse to main branch. parallel — independent stories: existing worktree lane. sequential — shared file scope or dep chain: run in order on main.
  • Falls back to count heuristic on API error or absent key — zero regression. Decision + reason logged to phase-cost.jsonl.

3.10 Idempotent Execution & Checkpointing (GAP-P13 Phase 1)

  • Each run gets a unique ORCH_RUN_ID. After each story completes, checkpoint_complete() appends an idempotency entry to logs/checkpoint-<phase>-<run-id>.jsonl.
  • On crash-restart with the same run ID, checkpoint_already_done() skips already-finished stories — preserving partial progress without re-running clean.
  • RESET_STORIES=true calls checkpoint_clear() before resetting PRD flags to ensure a true clean slate.

3.11 Structured Story Outputs (GAP-P17)

  • PRD stories can declare "outputSchema": {...} (JSON Schema). When present, claude.sh appends an output-schema instruction to the system prompt.
  • emit_story_artifact() extracts the first JSON object from the agent result and writes a StoryArtifact record to logs/story-artifacts.jsonl for every story (schema-less stories emit structuredOutput: null).

3.12 Sandbox Isolation (GAP-P14)

  • run-agent-orchestration.sh --sandbox wraps each agent in a Docker/Podman container built from Dockerfile.sandbox (node:20-slim + Claude CLI, non-root user).
  • lib/sandbox-invoke.sh replaces CLAUDE_CMD: mounts only PROJECT_ROOT r/w, sets --security-opt no-new-privileges, --cap-drop ALL, CPU + memory limits (default 2 CPUs / 4 GB).
  • Verified: writes inside PROJECT_ROOT succeed; writes outside (/etc/shadow) are blocked. Image auto-builds on first --sandbox run.
  • Env overrides: EPAM_SANDBOX_IMAGE, EPAM_SANDBOX_CPUS, EPAM_SANDBOX_MEMORY.

3.13 Full Pipeline Cost Tracking (GAP-P22)

  • run_orch_prompt() passes ORCH_JSON_RESULT to ai-run.sh, which captures the JSON response so cost/tokens can be extracted and written to phase-cost.jsonl via append_pipeline_cost_record().
  • Every pipeline agent is labeled: assessment, spec-coordinator, qa-gate:sast, qa-gate:spec-validator, qa-gate:review-ranger, qa-gate:mutant-hunter, qa-gate:fuzz-weaver, qa-gate:perf-sentinel, qa-gate:e2e, topology-router.
  • calibrate.py computes pipeline_overhead_ratio (EMA of pipeline/story cost ratio) stored in calibration.json. CPA output shows Total (est): $X.XX (incl. pipeline ×N.NN) alongside story estimate.
  • Scorecard shows separate "Avg Story Cost" and "Avg Total Cost" KPI cards.
  • First measured ratio: assessment agents dominate (~82% of pipeline). Full-gates hello-world pipeline = 7.75× story cost.

3.14 CPA Calibration & Blended Forecast Improvements

  • Model-aware buckets: calibration key is now effort:storyType:invokeMode:modelAlias — Haiku and Sonnet never share a bucket.
  • Blend floor: BLEND_ADJ_FLOOR=0.25 — adjusted estimate can't go below 25% of formula baseline.
  • Tighter interpolation: BLEND_LOW raised 0.50 → 0.60.
  • Impact: total blended variance improved from +233% to -14% on a clean 11-story run.

3.4 Logging

  • logs/agent-status.json — live orchestration state.
  • logs/agent-messages.jsonl — streaming transcripts.
  • logs/phase-cost.jsonl — token/cost ledger for all agents: story implementation (agent_type absent) and pipeline agents (agent_type set). GAP-P22 added pipeline tracking June 4, 2026.
  • logs/story-artifacts.jsonl — per-story structured output records (GAP-P17).
  • logs/provider-failover.json — sentinel for backup switches.

3.5 Specification Mode (OpenSpec/Speckit)

  • spec-mode-runner.js snapshots prd.json, invokes the coordinator agent, runs OpenSpec/Speckit per story, merges acceptance criteria, and appends JSONL records to logs/spec-phase.jsonl.
  • Artifacts: logs/spec-baseline.json, logs/spec-summary.json, and archived runs under logs/spec-runs/<run> so dashboards can diff baselines against current PRD.
  • CLI entrypoints: epam orchestrate spec <phase> (or REPL /orchestrate spec) and npm run specification:run -- --phase <phase>.
  • Toggle with EPAM_SPEC_MODE when you need to replay phases without rerunning spec agents.

4. Dashboards & Real-Time Updates

4.1 Eleventy Pipeline

npm run dashboards:build|watch|serve read orchestrations/dashboards/.eleventy.js, copying 11 HTML dashboards (monitor, PRD viewer, specification, etc.), runtime assets, PRD data, and selective logs into orchestrations/dashboards/live. Only required JSON/JSONL files are watched to limit rebuild noise.

4.1.1 Provider & Model Filters

monitor.html and prd-viewer.html expose dropdown filters for provider and model. Selecting a provider auto-populates the model dropdown from PROVIDER_MODELS config. A blank selection clears both filters and re-renders all rows. The filter state is kept in module-level variables so re-renders (triggered by the Eleventy watcher) preserve the selection.

4.2 Snapshot & Overlay

build/snapshot.js creates build-info.json (hashes, story counts, phase summaries). runtime/build-info.js polls that file in every dashboard, rendering a shared status pill and dispatching global events for page-specific scripts.

4.3 Integration with Orchestration

The master script starts the Eleventy watcher automatically, logs to dashboards-watch.log, and cleans up on exit. Generated assets are ignored by git but deployed to the demo workspace for validation.

4.4 Specification Dashboard

specification.html compares the baseline (logs/spec-baseline.json) with the current PRD, showing acceptance-criteria diffs, split stories, and the OpenSpec/Speckit ledger emitted by logs/spec-phase.jsonl. It pulls coverage metrics from logs/spec-summary.json and uses the shared runtime overlay for freshness warnings.

4.5 Scorecard Dashboard (GAP-P15)

scorecard.html aggregates phase-cost.jsonl, testing-gates.jsonl, and cpa-review.jsonl into a cross-run scorecard — no new instrumentation needed. Metrics: story pass rate, gate pass rate, first-attempt %, avg cost/story, avg time/story, CPA accuracy. Historical runs table with per-story detail for the most recent run.

4.6 SWE-bench Evaluation Dashboard (GAP-P3)

swe-bench.html displays results from the EPAM TypeScript Benchmark — 5 curated bug-fix tasks (logic, data structures, async, correctness). Score gauge, KPI strip, task table, run history. Results aggregated by syncBenchResults() in .eleventy.js from benchmarks/results/*.json. First run: 5/5 resolved (100%) at $0.40, avg 129 s/task. Run: bash scripts/run-swe-bench.sh (add --sandbox for containerized execution).

5. Deployment & Environments

5.1 Build Targets

  • npm run build — tsup produces dist/epam.js (CLI) and dist/sdk.js + dist/sdk.d.ts (library surface).
  • npm run dev — run entrypoint via tsx.
  • Quality: npm run lint, typecheck, test, dashboards:build.
  • Import as a library: import { AgentRunner, ProviderChain, createTools } from 'epam-cli/sdk'. See TOOL_REGISTRY.md for plugin authoring.

5.2 Demo Deployment

  1. Execute scripts/deploy-demo.sh.
  2. Script rebuilds, copies dist, syncs dashboards to epam-cli-demo, and verifies via node dist/epam.js --version.
  3. Demo workspace mirrors live dashboards for testing orchestration flows.

5.4 Quick-start & Keyless Demo (GAP-P18)

  • QUICKSTART.md — 3-step live demo (clone → set keys → run travel-app test).
  • demo/logs/ — canned snapshot of a completed Skyscanner orchestration run.
  • scripts/demo-mode.sh on/off — symlinks orchestrations/logs to demo/logs/ so all dashboards render without a live run. Restore with demo-mode.sh off.
  • Travel-app PRD includes SKY-001b (API discovery) as a scaffold prerequisite so the client is always implemented against a verified endpoint contract.

5.3 Environment Variables

  • CLI: EPAM_BACKEND_URL, EPAM_PROVIDER, EPAM_MODEL, EPAM_MAX_ITERATIONS, budget envs.
  • Secrets: EPAM_API_KEY_ANTHROPIC, EPAM_API_KEY_OPENAI, etc.
  • Orchestration: CLAUDE_CMD, EPAM_ORCHESTRATION_PROVIDER, EPAM_DASH_AUTO_SERVE, EPAM_REDIS_URL.
  • Specification: EPAM_SPEC_MODE (default 1) toggles the spec pre-pass.
  • Brownfield: EPAM_BROWNFIELD_ENABLED (default 1), EPAM_BROWNFIELD_MAX_FILES, EPAM_BROWNFIELD_MAX_TOKENS.
  • Webhooks: JIRA_WEBHOOK_SECRET, JIRA_BASE_URL, JIRA_API_TOKEN, JIRA_USER, CONTROL_PLANE_PORT (default 3001).
  • MCP: each server entry in .mcp.json accepts "enabled": false to skip at init time.
  • Sandbox: EPAM_SANDBOX_IMAGE (default epam-cli-sandbox:latest), EPAM_SANDBOX_CPUS (default 2), EPAM_SANDBOX_MEMORY (default 4g).
  • Orchestration durability: ORCH_RUN_ID — override for crash-resume (default: timestamp-based).

6. Operations

6.1 Failover & Token Policy

Follow plans/orchestration-failover-plan.md for forecasting tokens, precomputing provider/model tuples, wrapping agent executions with guards, and persisting failovers to logs/agent-status.json + phase-cost.jsonl.

6.2 Session Recovery

Use /resume for local JSONL sessions, /share//handoff for Redis export, and /import/epam import for rehydration. Future cross-process failover should call storeSession() and pass the ULID via EPAM_AUTO_RESUME.

6.3 Monitoring

Scripts like dashboard-health-check.sh, check-phase-gate.sh, and worktree-health-check.sh validate readiness. Tail orchestrations/logs for stuck lanes.

6.4 Automation Hooks

update-monitor.sh refreshes dashboards/alerts. provider-cutover.sh bulk-switches providers when a quota trips. sync-monitor-stories.sh keeps dashboards aligned with PRD changes.

8. Extension Playbook

Add Commands

Create src/cli/commands/<name>.ts, register it in src/cli/index.ts.

Add Providers

Implement LLMProvider, register slots in ProviderChain, add orchestration wrappers if needed.

Add Tools / Plugins

Built-ins: follow patterns under src/tools/builtin. External plugins: implement ToolPlugin (see TOOL_REGISTRY.md), name the package epam-tool-*, and list it in .epam/settings.json tools array. createTools() loads all plugins at startup.

Grow Dashboards

Add templates in orchestrations/dashboards, import the shared runtime overlay, and expose data via Eleventy data files.

Add Constitution Rules

Drop entries in .epam/constitution-rules.json with match.skills or match.agentRole predicates. Rules are injected per-story at runtime — no script changes needed.

Extend Brownfield Context

Tune EPAM_BROWNFIELD_MAX_FILES and EPAM_BROWNFIELD_MAX_TOKENS, or swap the TF-IDF ranker in lib/brownfield-context.js for semantic embeddings when EPAM_API_KEY_OPENAI is set.

Add Structured Outputs

Add "outputSchema": {...} (JSON Schema) to a PRD story. The agent appends a conforming JSON block to its response; emit_story_artifact() captures it in logs/story-artifacts.jsonl.

Add Benchmark Tasks

Create benchmarks/tasks/<id>/ with task.json (metadata + fail_to_pass test names), src/*.ts (buggy source), and src/*.test.ts (vitest tests). run-swe-bench.sh picks them up automatically.

Use as SDK

import { AgentRunner, ProviderChain, createTools } from 'epam-cli/sdk'. Full stable surface in src/sdk.ts; plugin authoring docs in TOOL_REGISTRY.md.
