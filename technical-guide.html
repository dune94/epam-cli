EPAM CLI Technical Guide

Comprehensive reference for the EPAM CLI runtime, agent architecture, orchestration scripts, dashboards, and operational tooling.

1. System Overview

Layer Responsibilities Key Entry Points

2. CLI Architecture

2.1 Entry & Commands

src/index.ts bootstraps Commander, injects chat when users only pass global flags, and parses asynchronously with helpful error output. src/cli/index.ts registers every command module so interactive and scripted flows reuse the same plumbing (auth, config, provider selection).

2.2 Config & Auth

  • ConfigResolver merges flags, EPAM_* env vars, project (.epam/settings.json), and global config (~/.epam/config.json), recording the model selection source for audit trails.
  • AuthManager runs device/browser login flows, persists secrets in the OS keychain, and backs epam provider * commands.

2.3 Session Lifecycle

  • SessionStore writes turns to .epam/sessions/<ulid>.jsonl for resume/fork/import flows.
  • RedisSessionStore (via EPAM_REDIS_URL) powers /share and /handoff by storing serialized bundles with a 7-day TTL.

2.4 REPL UX

Repl coordinates provider failover, slash commands, streaming output, and history. RawInputBox draws the prompt, supports bracketed paste, and keeps cursor math accurate for multi-line input.

2.5 Agent Execution

  1. chat builds the system prompt (constraints + consultation context).
  2. AgentRunner maintains conversation state and token ceilings.
  3. Tool calls go through ToolRunner (Read/Write/List/Search/Fetch/Bash).
  4. RalphWiggumLoop retries fragile bash steps automatically.
  5. AuditorRunner feeds lint/test findings back into the chat.

2.6 Provider Chain

ProviderChain spins up slots for Anthropic, OpenAI, Gemini, Copilot, Codex, Cursor, Qwen, Codemie, and Proxy providers. It tracks slot health, classifies errors (retry vs. failover), and emits events consumed by the REPL. Backups follow the policy documented in plans/orchestration-failover-plan.md so CLI and orchestration share the same strategy.

2.7 Batch Commands

epam run executes a single prompt non-interactively but still uses the agent pipeline. estimate, orchestrate, phase, report, etc. wrap orchestration scripts to keep tooling consistent.

3. Orchestration Architecture

3.1 Story Data

  • orchestrations/prd.json — backlog, phases, quotas.
  • orchestrations/agents/profiles.json — persona prompts.
  • AGENTS.md, KB.md — learned behaviors & knowledge base.

3.2 Script Flow

  1. Bootstrap paths, pick provider wrapper, ensure logs.
  2. Start Eleventy watcher (unless EPAM_DASH_AUTO_SERVE=0).
  3. Run specification pre-pass (spec-mode-runner.js) — snapshot prd.json, assign OpenSpec/Speckit per story, apply acceptance criteria edits or splits, and log deltas to logs/spec-phase.jsonl (disable with EPAM_SPEC_MODE=0).
  4. Resolve phase + orchestration mode via phasesConfig.
  5. Run CPA + execution-plan jq pipelines.
  6. Forecast token ceilings, precompute provider/model backups.
  7. Create git worktrees per lane and seed manifests.
  8. Launch primary/independent agents via provider wrapper scripts.
  9. Update dashboards + logs, trigger provider-cutover.sh as needed.
  10. Run reviews, tear down worktrees, stop watcher.

3.3 Utilities

Wrapper scripts (claude.sh, copilot.sh, openai.sh, etc.) normalize provider CLIs. Supporting helpers include contextualize-stories.sh, estimate-stories.sh, team-lead-review.sh, update-monitor.sh, and provider-cutover.sh.

3.4 Logging

  • logs/agent-status.json — live orchestration state.
  • logs/agent-messages.jsonl — streaming transcripts.
  • logs/phase-cost.jsonl — token/cost ledger per story/provider.
  • logs/provider-failover.json — sentinel for backup switches.

3.5 Specification Mode (OpenSpec/Speckit)

  • spec-mode-runner.js snapshots prd.json, invokes the coordinator agent, runs OpenSpec/Speckit per story, merges acceptance criteria, and appends JSONL records to logs/spec-phase.jsonl.
  • Artifacts: logs/spec-baseline.json, logs/spec-summary.json, and archived runs under logs/spec-runs/<run> so dashboards can diff baselines against current PRD.
  • CLI entrypoints: epam orchestrate spec <phase> (or REPL /orchestrate spec) and npm run specification:run -- --phase <phase>.
  • Toggle with EPAM_SPEC_MODE when you need to replay phases without rerunning spec agents.

4. Dashboards & Real-Time Updates

4.1 Eleventy Pipeline

npm run dashboards:build|watch|serve read orchestrations/dashboards/.eleventy.js, copying 11 HTML dashboards (monitor, PRD viewer, specification, etc.), runtime assets, PRD data, and selective logs into orchestrations/dashboards/live. Only required JSON/JSONL files are watched to limit rebuild noise.

4.2 Snapshot & Overlay

build/snapshot.js creates build-info.json (hashes, story counts, phase summaries). runtime/build-info.js polls that file in every dashboard, rendering a shared status pill and dispatching global events for page-specific scripts.

4.3 Integration with Orchestration

The master script starts the Eleventy watcher automatically, logs to dashboards-watch.log, and cleans up on exit. Generated assets are ignored by git but deployed to the demo workspace for validation.

4.4 Specification Dashboard

specification.html compares the baseline (logs/spec-baseline.json) with the current PRD, showing acceptance-criteria diffs, split stories, and the OpenSpec/Speckit ledger emitted by logs/spec-phase.jsonl. It pulls coverage metrics from logs/spec-summary.json and uses the shared runtime overlay for freshness warnings.

5. Deployment & Environments

5.1 Build Targets

  • npm run build — tsup compile to dist/epam.js.
  • npm run dev — run entrypoint via tsx.
  • Quality: npm run lint, typecheck, test, dashboards:build.

5.2 Demo Deployment

  1. Execute scripts/deploy-demo.sh.
  2. Script rebuilds, copies dist, syncs dashboards to epam-cli-demo, and verifies via node dist/epam.js --version.
  3. Demo workspace mirrors live dashboards for testing orchestration flows.

5.3 Environment Variables

  • CLI: EPAM_BACKEND_URL, EPAM_PROVIDER, EPAM_MODEL, EPAM_MAX_ITERATIONS, budget envs.
  • Secrets: EPAM_API_KEY_ANTHROPIC, EPAM_API_KEY_OPENAI, etc.
  • Orchestration: CLAUDE_CMD, EPAM_ORCHESTRATION_PROVIDER, EPAM_DASH_AUTO_SERVE, EPAM_REDIS_URL.
  • Specification: EPAM_SPEC_MODE (default 1) toggles the spec pre-pass.

6. Operations

6.1 Failover & Token Policy

Follow plans/orchestration-failover-plan.md for forecasting tokens, precomputing provider/model tuples, wrapping agent executions with guards, and persisting failovers to logs/agent-status.json + phase-cost.jsonl.

6.2 Session Recovery

Use /resume for local JSONL sessions, /share//handoff for Redis export, and /import/epam import for rehydration. Future cross-process failover should call storeSession() and pass the ULID via EPAM_AUTO_RESUME.

6.3 Monitoring

Scripts like dashboard-health-check.sh, check-phase-gate.sh, and worktree-health-check.sh validate readiness. Tail orchestrations/logs for stuck lanes.

6.4 Automation Hooks

update-monitor.sh refreshes dashboards/alerts. provider-cutover.sh bulk-switches providers when a quota trips. sync-monitor-stories.sh keeps dashboards aligned with PRD changes.

7. Quick Reference

CLI

  • epam chat — REPL with resilient provider chain.
  • epam run — deterministic single-run agent.
  • epam estimate|orchestrate|phase ... — script wrappers.
  • epam orchestrate spec <phase> — coordinator → OpenSpec/Speckit pre-pass.
  • epam provider * — provider auth lifecycle.
  • epam import <code> — hydrate Redis/shared sessions.

Orchestration

  • orchestrations/scripts/run-agent-orchestration.sh
  • orchestrations/scripts/spec-mode-runner.js
  • .../provider-cutover.sh, .../update-monitor.sh
  • scripts/deploy-demo.sh — keep demo env current.

Dashboards

  • npm run dashboards:build — static build.
  • npm run dashboards:serve — Eleventy + BrowserSync.
  • Watcher auto-started by orchestration runs (toggle with EPAM_DASH_AUTO_SERVE).

8. Extension Playbook

Add Commands

Create src/cli/commands/<name>.ts, register it in src/cli/index.ts.

Add Providers

Implement LLMProvider, register slots in ProviderChain, add orchestration wrappers if needed.

Add Tools/Auditors

Follow patterns under src/tools/builtin and src/auditors, wire them into the tool resolver.

Grow Dashboards

Add templates in orchestrations/dashboards, import the shared runtime overlay, and expose data via Eleventy data files.
